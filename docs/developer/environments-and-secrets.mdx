# Environments and Secrets

This page maps the environment variables that matter for launch and the non-negotiable production checks already enforced by code.

## API environment groups

The API is the strictest service in the stack. Its `.env.example` and `app/config.py` define the launch-critical contract.

### Security and identity

- `SECRET_KEY`: required and must be at least 32 characters in production.
- `ENCRYPTION_KEY`: strongly recommended for production key protection.
- `ENCRYPTION_KEY_PREVIOUS`: optional decrypt-only fallback during key rotation.
- `ADMIN_API_KEY`: required if you use admin routes; treat 32+ chars as the minimum bar.
- `CLERK_JWKS_URL`: required for Clerk token verification in any real auth environment.
- `CLERK_AUDIENCE`, `CLERK_ISSUER`, `DEV_CLERK_*`: optional claim controls and local-dev support.

### Data plane

- `DATABASE_URL`
- `DATABASE_URL_INGEST`
- `DATABASE_URL_APP`
- `DATABASE_URL_EVIDENCE`
- `DATABASE_URL_ADMIN`
- `REDIS_URL`

Production code already rejects `REDIS_URL=redis://localhost:6379/0` when `ENVIRONMENT=production`.

### Customer-facing integration

- `CORS_ORIGINS`: must include the real dashboard and marketing domains.
- `FRONTEND_URL`: used for redirect flows such as billing.
- `API_BASE_URL`: used for server-generated callback links.

### Storage and imports

- `R2_ENDPOINT`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET`

If import workflows or evidence pack storage are part of launch, configure R2 before go-live.

### Billing and monetization

- `STRIPE_SECRET_KEY`
- `STRIPE_WEBHOOK_SECRET`
- `STRIPE_PRICE_*`

### Optional but production-useful

- `GLITCHTIP_DSN`
- `OPENAI_API_KEY`
- `GEMINI_API_KEY`
- `GOOGLE_API_KEY`
- `ANTHROPIC_API_KEY`

## Dashboard environment groups

`trustscope-dashboard` needs both public browser configuration and server-side secrets for proxy routes.

### Browser-visible

- `NEXT_PUBLIC_API_URL`
- `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`
- `NEXT_PUBLIC_CLERK_SIGN_IN_URL`
- `NEXT_PUBLIC_CLERK_SIGN_UP_URL`
- `NEXT_PUBLIC_CLERK_SIGN_IN_FALLBACK_REDIRECT_URL`
- `NEXT_PUBLIC_CLERK_SIGN_UP_FALLBACK_REDIRECT_URL`

### Server-only or operator-only

- `CLERK_SECRET_KEY`
- `TRUSTSCOPE_API_URL`
- `ADMIN_API_KEY`
- `PREVIEW_TOKEN`
- `NEXT_PUBLIC_DEV_TIER` for local tier forcing only

`vercel.json` currently pins `NEXT_PUBLIC_API_URL=https://api.trustscope.ai` for the deployed dashboard.

## Website environment groups

The marketing site is mostly static. The only clear runtime dependency in code is the waitlist bridge:

- `API_URL`

If unset, the site falls back to `https://api.trustscope.ai`.

## Admin environment groups

The admin console uses:

- `API_URL`
- `ADMIN_API_KEY`

If the admin app is part of launch operations, move these into the hosting provider's secret store and document the hosting target explicitly.

## Docs environment groups

The docs site does not require secrets to build, but the current Docusaurus config includes placeholder Algolia settings:

- `appId: YOUR_APP_ID`
- `apiKey: YOUR_SEARCH_API_KEY`

Before public launch, either configure real search credentials or disable/search-hide the integration so the docs do not ship a broken search box.

## Production invariants

Use these as launch blockers if they are not satisfied:

1. API `SECRET_KEY` is strong and rotated out of any development default.
2. API `REDIS_URL` and `DATABASE_URL` point to production infrastructure, not localhost.
3. Clerk live keys and JWKS settings are loaded in both API and dashboard.
4. Dashboard and API domains are aligned in `CORS_ORIGINS`.
5. Stripe live keys and webhook secret are configured if billing is live on day one.
6. R2 credentials exist if import and evidence-pack workflows are live on day one.

## Secret management guidance

- Store runtime secrets in Render and Vercel environment settings, not in repo files.
- Use separate development, preview, and production values.
- Rotate any key that has ever been committed, copied into a test doc, or exposed in a public demo.
- Keep a short ownership list for Clerk, Stripe, Render, Vercel, and DNS so emergency rotation is not blocked by access confusion.
